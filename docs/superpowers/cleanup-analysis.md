# tt-station — cleanup / simplification analysis

*Read-only analysis. Written 2026-07-03. Scope: `crates/`, `macos/TTStation/`, `docs/`.*

## Health summary

For a project built in a handful of subagent-driven sessions against real, only-just-discovered
hardware behavior, the codebase is in unusually good shape: every non-trivial decision is
documented in place (arguably *over*-documented — comment-to-code ratio is high, which is why
several "oversized" files below are large in bytes but not in actual logic), tests are
comprehensive, and the team has a strong habit of leaving a paper trail (`.superpowers/sdd/`,
`CLAUDE.md`'s deferred-tickets list) for exactly the kind of debt this report catalogs. The real
cruft is narrow and traceable to one root cause — **rapid, real-hardware-driven iteration outpaced
doc/test upkeep in a few specific spots**: two self-acknowledged duplicated test fixtures, one
dead client method left behind after a call site moved, one still-open serde footgun with a
workaround instead of a fix, a couple of stale doc claims (`docs/client-agent-integration-findings.md`
lists two findings as open that are already fixed in code; the macOS README describes v2-deferred
features as if in scope), and one asymmetry worth watching — `DockerBackend` is a well-labeled
fallback that has quietly fallen behind `RunPyBackend` on every real-hardware fix (device reset,
stale-container cleanup, model-id reconciliation) discovered since it was written. None of this is
alarming; most of it is Quick-win-sized.

## Notes on what's *not* cruft

- `DstackBackend` (`crates/tt-station-agentd/src/serving/dstack.rs`) is a small, clearly-labeled
  intentional stub ("M4... this stub exists purely so the trait has two implementations from day
  one") with tests that pin its documented behavior. Nothing to clean up there.
- The docker-vs-runpy split, the `ServingBackend` trait boundary, and the extensive per-field doc
  comments throughout `routes.rs`/`runpy.rs`/`docker.rs` are a large share of those files' line
  counts but are genuinely load-bearing institutional memory (e.g. exact real-hardware error
  strings that justified a fix) — the *decomposition* suggestions above preserve all of it, just
  in smaller files, not compress or delete it.
